# Revendor to Gardener v1.146.4 — Design

**Date:** 2026-07-17
**Branch:** `odp/gardener-revendor`
**Baseline:** gardener v1.144.1 (VERSION `v0.7.0-odp.2`)
**Target:** gardener v1.146.4

## Goal

Move `gardener-extension-provider-hcloud` to gardener v1.146.4 and bring every other
dependency to a current, mutually-consistent, supported version.

This resolves upstream issue [#310](https://github.com/23technologies/gardener-extension-provider-hcloud/issues/310)
("No released version runs on Kubernetes >= 1.34 seeds") and is the point at which the
`odp/v0.6.42-k8s-1.35` stopgap fork (tag `v0.6.42-odp.1`) can be retired.

## What "up to date" means here

For a Gardener extension, "current" mostly means **match gardener's pins**, not **take latest**.
Gardener v1.146.4's go.mod is the backbone; diverging from it breaks alignment with core.

Verified from `raw.githubusercontent.com/gardener/gardener/v1.146.4/go.mod`, this repo already
matches on every line that matters:

| Pin | gardener v1.146.4 | this repo | action |
|---|---|---|---|
| `k8s.io/*` (staging set) | v0.35.5 | v0.35.5 | hold |
| `sigs.k8s.io/controller-runtime` | v0.23.3 | v0.23.3 | hold |
| `k8s.io/autoscaler/vertical-pod-autoscaler` | v1.6.0 | v1.6.0 | hold |
| `github.com/gardener/etcd-druid/api` | v0.36.4 | v0.36.4 | hold |
| `k8s.io/utils` | `v0.0.0-20260507154919-ff6756f316d2` | identical | hold |
| `go` directive | 1.26.0 (floor) | 1.26.2 | hold |

**The traps:** controller-runtime v0.24.1 and VPA v1.7.0 each require `k8s.io/* v0.36.x` and would
silently drag the whole staging set off gardener's pin. Renovate PRs #213 and #307 propose exactly
these, and `.github/renovate.json` sets `automerge: true` for gomod — so the hold strategy currently
has *negative* enforcement. Commit 7 fixes that.

## Scope of change (verified, not assumed)

gardener v1.144.1 → v1.146.4 is **143 commits**, touching 22 of the 83 gardener packages this repo
imports. Diffed against the real module source in `GOMODCACHE` (the GitHub compare API caps its file
list at 300 and cannot be trusted here):

- **`extensions/` — the API this repo codes against: only `pkg/controller/worker` changed.**
  That is `WorkerPoolHash` (§ commit 4). Nothing else in the extension library moved.
- **`third_party/open-telemetry/opentelemetry-operator/apis/{v1alpha1,v1beta1}` were removed.**
  Transitive only — this repo has no direct imports, and v1.146.4 replaces the vendored copy with
  the real `github.com/open-telemetry/opentelemetry-operator/apis v0.153.0` module. Resolves
  automatically.
- The other 19 are gardener-internal. The compiler and `make generate` are the gate.

## Commit sequence

The build is intentionally red between commits 1 and 4: the tidy fix must precede tidy, and the
compile fix follows the revendor.

**1 — unblock `go mod tidy`.** `hack/tools.go`: delete `_ "github.com/gardener/gardener/hack"`
(package removed at v1.146.0; verified HTTP 404 at v1.146.4, 200 at v1.144.1) and
`_ "github.com/ironcore-dev/vgopath"`.
Keep `_ "k8s.io/code-generator"` — it is the only thing holding code-generator as a direct require,
which `update-codegen.sh:14` reads. Keep `_ ".../hack/tools"` — load-bearing for `tools.mk`.

**2 — codegen migration (the release blocker).** v1.146.4 deletes `hack/vgopath-setup.sh` (verified
404), which `hack/update-codegen.sh:12` sources under `set -o errexit`. Remove that line *and* the
now-orphaned `rm -f $GOPATH/bin/*-gen` on line 17 — with the source gone, `$GOPATH` is unset under
`set -o nounset`, so removing only the first line still fails. Drop `$(VGOPATH)` from the Makefile
`generate` target. Mirrors `gardener-extension-provider-aws`, which has already completed this.

Why this matters: `.github/workflows/on-release-publish.yml` runs `make generate` to produce
`example/controller-registration.yaml`. Without this fix the branch compiles clean and is
**unreleasable** — failure appears only at tag time.

**3 — revendor.** `gardener` **and** `gardener/pkg/apis` → v1.146.4, both explicit. Core's go.mod
pins `pkg/apis v1.144.2` behind a `replace` that does not propagate downstream, so dropping the
explicit require silently resolves to v1.144.2 *and still compiles*. Also MCM → v0.62.1,
hcloud-go → v2.44.0. `go mod tidy` auto-adds `hack/tools v1.146.4`, raises ginkgo/gomega, drops
controller-tools, and raises `x/net` → v0.56.0 (clears 7 CVEs).

**etcd-druid stays untouched.** `api@v0.36.4` has no `v1alpha1/` package (verified: it ships
`common/ config/ core/ validation/`), and `pkg/cmd/controller/cmd.go:25` resolves through the
**root** module at v0.27.0. Removing the require breaks the build.

**4 — compile fix.** `WorkerPoolHash` went 5 args → 4 at v1.146.0. `pkg/controller/worker/machines.go:155`
passes 5 → `worker.WorkerPoolHash(pool, w.cluster, nil, nil)`.

**5 — 2026-10-01 deadline.** `pkg/controller/worker/actuator_hooks.go` reads `Datacenter.ServerTypes`,
which Hetzner stops returning after 2026-10-01. Move to `ServerType.Locations`. Also stop discarding
the error returns (`dc, _, _ :=`) so an API break surfaces as an API error rather than a bogus
"Machine Type not available" for every pool.

**6 — images + builder.** CCM → v1.33.0, csi-driver → v2.22.0, `golang:1.26.5`.
Sidecars: hold attacher/resizer/provisioner on Hetzner's tested matrix (csi-driver v2.22.0's chart
pins them exactly, and shipped *after* the newer ones existed — deliberate). Take
`csi-node-driver-registrar` v2.17.0 and `livenessprobe` v2.19.0: pure CVE fixes, zero flag/RBAC
delta, and neither touches the Kubernetes API.

**7 — Renovate guardrails.** `allowedVersions` ceilings — k8s group `<0.36.0`, controller-runtime
`<0.24.0`, controller-tools `<0.21.0`, VPA `<1.7.0` — and `automerge: false` for gomod.

## Verification

**e2e cannot validate this.** The nightly harness has been structurally dead since the *current*
revendor: `hack/ci/00-environment.sh` seds `example/provider-extensions/` paths that return 404 at
v1.144.1+, under `set -eu`. Every k8s version it pins (1.26–1.28) is below gardener's 1.32 floor.
Rebuilding it is its own project.

So the gates are local, in order:

1. `go mod tidy` clean
2. `go list -m` assertions — `gardener` **and** `gardener/pkg/apis` both v1.146.4 (catches the
   silent skew), `k8s.io/api` v0.35.5, `controller-runtime` v0.23.3, VPA v1.6.0
3. `go build ./...`
4. `go vet ./...`
5. **`make generate` + diff the result** — the real release gate; the post-vgopath codegen path is
   otherwise unexercised
6. `make test`
7. `make check`
8. **No-rollout proof for `WorkerPoolHash`** — hash must be byte-identical for the `nil`-additionalData
   call, or every worker node rolls. Verify by reading both implementations end-to-end rather than
   trusting the reasoning.

## Kubernetes support after this

`SupportedVersions` at v1.146.4 is 1.32–1.36 (verified in
`pkg/utils/validation/kubernetesversion/version.go`). The **binding floor is 1.33**, set by Hetzner
CCM/CSI (which dropped 1.32), not by gardener — and that floor was already in effect before this work.

Known exposure, pre-existing and not introduced here: `csi-provisioner` v6.2.0 declares min k8s
**1.34**, and no shoot-version gating exists around the CSI sidecars. Worth a decision by whoever
owns the supported-shoot matrix.

## Follow-ups — deferred until commits 1–7 land

Explicitly out of scope for this PR. Each gets its own PR, in roughly this order:

1. **distroless debian13** — debian12 EOLs Sept 2026 (~2 months). debian13 verified available,
   nonroot UID 65532 unchanged, static `CGO_ENABLED=0` binary has no glibc coupling. Note gardener
   v1.146.4 is still on debian12, so this steps ahead of upstream. Lowest-risk change in the set.
2. **CI Actions sweep** — node12-era bundles (checkout v2, setup-go v2, docker/* v1–v2) running on
   Node 24. Dated cliff: **2026-09-16**, when Node 20 is removed from runners. Standardize checkout
   across all **5** call sites (`on-release-publish.yml` pins it twice — miss one and it silently
   stays on v3). Replace `go-version: ">=1.21.0"` with `go-version-file: go.mod`. Then add
   `"github-actions"` to `enabledManagers` as the ratchet — its omission is precisely why these
   pins rotted while gomod stayed current. **Must land after the Renovate guardrails (commit 7).**
3. **containerd pin** — current `Pin: version 1.7.*` in `pkg/webhook/controlplane/ensurer.go` is
   likely a **silent no-op on jammy** (jammy publishes only 1.5.9 and 2.2.1 — no 1.7.x — so apt
   installs the broken 2.2.1 anyway; noble does have 1.7.12). Proposed: negative pin
   `Pin: version 2.2.[01]*` / `Pin-Priority: -1`, which self-clears when Ubuntu backports 2.2.2.
   Verify with `apt-cache policy containerd` on a real jammy node first. Strategic: containerd 1.7
   is EOL Sept 2026 and k8s 1.35 is the last release supporting containerd 1.x — any 1.7 pin is
   short-lived either way.
4. **etcd-druid `api/core/v1alpha1` migration** — repoint `cmd.go:25`, promote `api` to direct at
   v0.36.4 (gardener's pin, not v0.37.0), drop the root require. A real package move affecting
   scheme registration, not a path swap.
5. **`.golangci.yaml` v1→v2 migration + 10 latent lint issues.** **Pre-existing, confirmed not a
   regression:** gardener v1.144.1's `tools.mk` pins `GOLANGCI_LINT_VERSION ?= v2.12.2` — the same
   version v1.146.4's `hack/tools` pins — and that linter rejects the config on its own
   (`unsupported version of the configuration`). So `make check` has been red since before this work.
   `run.deadline` is not even valid v1 anymore (removed for `timeout` in v1.57), so `golangci-lint
   migrate` refuses until `deadline`→`timeout` is fixed by hand first; after that, `migrate` produces
   a clean v2 config that verifies.
   Doing so surfaces **10 pre-existing staticcheck issues** in code untouched by the revendor:
   `pkg/controller/worker/machines.go:31-32` (same MCM package imported twice, as `machinev1alpha1`
   and `mcmv1alpha1`), `pkg/hcloud/apis/transcoder/cloud_profile.go` (×4), `pkg/hcloud/apis/validation/`
   (×2, incl. `sets.String` deprecated → `sets.Set[string]`), `pkg/controller/controlplane/valuesprovider.go:346`
   (Yoda condition), `pkg/webhook/validator/cloudprofile.go:47`. Not urgent: **CI never runs
   `make check`** (`on-pr-test.yml` runs `make install` + `make test-cov` only).

6. **MCM-provider-hcloud `datacenter` bug — its own incident, not this plan.**
   `pkg/hcloud/machine_controller.go` sends the `datacenter` field that Hetzner removed from the
   server-create API on **2026-07-01**. Already live-broken, independent of this upgrade, in an
   unmaintained external repo (v0.3.1 is 13 months stale) with no fix available. Failure mode —
   hard 400 vs **silently ignored** — is inference, not documented; "silently ignored" is the
   worse outcome (servers land in arbitrary locations, zone spreading degrades invisibly).
   Test against the live API. Fix is small (`GetRegionFromZone` already exists; set `.Location`
   instead of `.Datacenter` — `Validate()` rejects both at once, so it must be a swap) but needs
   a fork + release.

  (#320, #321) that drift *off* gardener's pin. This branch is 12 ahead. Separate decision.
